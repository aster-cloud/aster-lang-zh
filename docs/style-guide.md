# Aster Lang 中文样式指南

本文档说明 Aster Lang 中文（zh-CN）的**正确编写方式**和**设计原则**。
完整决策背景见 [ADR-0008](../../aster-cloud/docs/architecture/decisions/0008-zh-cn-v2-keywords-and-punctuation.md)。

## 一句话概括

> **Aster 中文是一门"形式语言 + 中文皮肤"，不是"自然中文"。** 关键字必须用空格分隔，
> 但可以使用中文标点（，。：；、）作为自然节奏；中文标点会自动归一化为英文等价。

## 标点使用

### 推荐：用中文标点写代码（自然）

```aster
模块 业务.贷款。

定义 申请 包含 客户编号，金额，年龄。

规则 评估 给定 申请，产出：
  如果 申请.年龄 小于 18：
    返回 「未成年」。
  返回 「合格」。
```

### 也接受：用英文标点（与英文 Aster 完全一致）

```aster
模块 业务.贷款.

定义 申请 包含 客户编号, 金额, 年龄.

规则 评估 给定 申请, 产出:
  如果 申请.年龄 小于 18:
    返回 「未成年」.
  返回 「合格」.
```

两种写法**完全等价**——canonicalizer 在编译第一步就把中文标点替换为英文等价。

### 标点映射表

| 中文 | 英文 | 用途 |
|---|---|---|
| `。` | `.` | 语句终止符 |
| `：` | `:` | 块起始符 |
| `，` | `,` | 列表/字段分隔符 |
| `；` | `;` | 块内分隔 |
| `、` | `,` | 枚举分隔（与逗号等价） |

### 字符串字面量内的标点保留

```aster
返回 「张三，李四：请于明天到岗。」。
```

字符串内的 `，`、`：`、`。` **100% 保留**——它们是用户数据，不是代码结构。

## 关键字使用

### 推荐子集（业务最常用）

```
模块 / 定义 / 包含 / 规则 / 给定 / 产出 / 返回 /
如果 / 否则 / 匹配于 / 当 /
令 / 定义为 / 将 / 设为 /
等于 / 大于 / 小于 / 不足 / 超过 / 多于 / 至少 / 至多 /
或者 / 并且 / 不是 /
加上 / 减去 / 乘以 / 除以 /
真值 / 假值 / 空值 / 无值 / 有值 / 成功值 / 失败值 /
文本 / 整数 / 小数 / 布尔 /
对每个 / 属于 / 必填 / 介于
```

### 设计原则：所有关键字都是 2+ 字

中文常用 1 字虚词（或、且、非、是、在、为、加、减、乘、真、假、空、无）
**永远不会被识别为关键字**——它们是**完全合法的标识符片段**。

实际意义：

```aster
// 这些写法 100% 安全：
定义 客户 包含 或然率，真客户标识，是否成年，在职状态，和约编号。
令 真实姓名 定义为 客户.真客户标识。
如果 客户.是否成年 等于 真值：
  返回 真值。
```

如果用 v1 关键字（已废弃），上面这段会被错误分词。v2 设计的**核心目标**就是
让业务术语永远优先于关键字。

## 块结构

Aster 中文使用缩进 + 冒号定义代码块（与 Python 一致）：

```aster
规则 评估 给定 申请，产出：     ← 块起始：冒号
  如果 申请.金额 大于 100000：  ← 2 空格缩进 + 冒号
    返回 「金额过大」。         ← 嵌套块
  返回 「通过」。               ← 退出嵌套
```

**强制 2 空格缩进**（与 en-US 一致）。tab 字符自动转换。

## 标识符命名

- **支持中文标识符**：变量名、函数名、字段名、类型名都可以是纯中文
- **不强制中文/英文一致**：可以混用（`客户Id`、`appName`、`货号` 共存）
- **建议**：业务领域词汇用中文（增加可读性），通用编程概念用英文（方便复用）

### 推荐示例

```aster
// 业务领域用中文
定义 保险申请 包含
  申请人，
  车辆，
  保额。

// 通用工具函数用英文
规则 formatDate 给定 date，产出：
  返回 date.toISOString。
```

## 字符串引号

中文 Aster 使用**直角引号** `「」`：

```aster
返回 「成功」。     ← 推荐
返回 "success".   ← 也可以，但不建议混用
```

智能引号 `""""` 会自动归一化为 `「」`。

## 函数定义两种风格

### 风格 1：函数式（参数表清晰）

```aster
规则 计算保费 给定 申请，产出 数值：
  返回 申请.车辆.价格 乘以 0.05。
```

### 风格 2：自然语言糖（可选）

```aster
评估 入参 申请，产出：
  返回 「合格」。
```

两种风格在编译阶段等价。**业务代码建议用风格 1**——结构更明确，工具友好。

## 注释

```aster
// 单行注释
# 也支持 # 风格的注释
模块 测试.示例。
```

注释中的中文 100% 保留——它们是给人看的，不是代码。

## 常见反模式（**不要这样写**）

### ❌ 不要省略空格

```aster
// 错误：关键字之间没有空格
规则评估给定申请产出：
```

正确：

```aster
规则 评估 给定 申请，产出：
```

理由：Aster 是**形式语言**，不是自然中文。强制空格是为了机器可读，
跟"美观与否"无关。

### ❌ 不要混用引号

```aster
// 错误：开头「，结尾"，不配对
返回 「成功"。
```

### ❌ 不要在标识符里嵌入关键字字符（除非有意如此）

```aster
// 容易混淆，但**技术上合法**：
定义 X 包含 包含。       // 字段名"包含"与关键字"包含"同字
```

这在 v2 下**仍然可以编译通过**（lexer 输出 token，parser 决定语义），
但**强烈建议**用更明确的字段名（如 `包含项`、`含有`）避免读者困惑。

## 设计哲学：为什么砍掉 1 字关键字？

v1 lexicon 把 `或` `且` `真` `假` 等单字作为关键字。这是经典的语言设计错误：

**中文常用字几乎都不能作为关键字**——因为它们高频出现在业务术语中。
`或然率`、`真客户`、`是否成年` 这种合法业务标识符会**直接撞上**单字关键字。

V2 通过把 1 字关键字升级为 2+ 字（`或者`/`并且`/`真值`/`假值`），
完全消除了这类冲突。代价是关键字稍长——但**关键字比标识符少得多**，
整体代码可读性反而提升。

## 与英文 Aster 的关系

中文 Aster 和英文 Aster **共享同一 AST**。canonicalizer 把中文关键字翻译
为英文 IR，然后走同一个 parser。这意味着：

- 中文写的策略，**可以**翻译为英文展示
- 英文写的策略，**可以**翻译为中文展示
- 同一个策略的 AST、执行结果、性能特征都相同

但**不建议**在同一个项目中混用中英文源码——见 ADR-0008 的"多语言投影"讨论。
